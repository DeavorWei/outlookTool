### C1. `handleActionQuit` 使用 `os.Exit(0)` 绕过所有 defer 清理

| 属性 | 内容 |
|------|------|
| **来源** | 报告1 H5、报告3 C2 |
| **位置** | [server.go:341-355](file:///d:/Document/GO/outlookTool/internal/server/server.go#L341-L355) |
| **核查结论** | ✅ 属实 |

**问题描述**：Web 界面的"退出"功能在 500ms 延迟后直接调用 `os.Exit(0)`，跳过 main.go 中所有 defer 清理，包括：
- `defer release()` —— 系统互斥锁不释放
- `defer zlog.Sync()` —— 日志不刷写
- `defer sched.Stop()` —— 调度器不停止
- `defer ole.CoUninitialize()` —— COM 资源不反初始化

而托盘菜单的退出走的是正常的 ctx 取消流程，两条路径行为不一致。

**修复方案**：
```go
// server.go - handleActionQuit 改为通过回调触发优雅退出
type WebServer struct {
    // ... 现有字段
    onShutdown func() // 新增退出回调
}

func (ws *WebServer) handleActionQuit(w http.ResponseWriter, r *http.Request) {
    logger.Info("通过 Web 界面触发退出")
    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(map[string]string{"status": "exiting"})
    if f, ok := w.(http.Flusher); ok {
        f.Flush()
    }
    // 通过回调触发与托盘相同的优雅退出流程
    go ws.onShutdown()
}
```

### C2. COM 工作线程 `Run` 循环无 panic recovery，单次 panic 导致进程崩溃

| 属性 | 内容 |
|------|------|
| **来源** | 报告1 L5、报告3 C5 |
| **位置** | [bridge.go:33-50](file:///d:/Document/GO/outlookTool/internal/outlook/bridge.go#L33-L50) |
| **核查结论** | ✅ 属实 |

**问题描述**：所有 COM 操作在该单线程执行。`task()` 执行时没有 `defer recover()` 保护。如果任何 COM 操作 panic（如损坏邮件的 `Move`、空指针 `IDispatch`），整个程序立即崩溃。对于长年无人值守的守护进程，这是致命稳定性缺陷。

**修复方案**：
```go
// bridge.go - Run 方法中为每个 task 包装 recover
for {
    select {
    case task := <-b.taskCh:
        func() {
            defer func() {
                if r := recover(); r != nil {
                    logger.Error("COM 任务 panic",
                        zap.Any("panic", r),
                        zap.String("stack", string(debug.Stack())))
                    // 让调用方收到 error 而非死等
                    // task 内部的 errCh 已由 panic 导致不可写，
                    // 需要在 Submit 中额外处理（见 C3 修复）
                }
            }()
            task()
        }()
    case <-ctx.Done():
        return
    }
}
```

### C3. `COMBridge.Submit` 在 context 取消时永久阻塞（goroutine 泄漏/死锁）

| 属性 | 内容 |
|------|------|
| **来源** | 报告2 第2点、报告3 C6 |
| **位置** | [bridge.go:53-64](file:///d:/Document/GO/outlookTool/internal/outlook/bridge.go#L53-L64) |
| **核查结论** | ✅ 属实 |

**问题描述**：
1. `Submit` 没有 `ctx` 参数，无法感知上下文取消
2. 如果 `Run` 因 `ctx.Done()` 退出，已入队但未执行的任务永远不会写入 `errCh`
3. 调用方在 `<-errCh` 上永久阻塞，造成 goroutine 泄漏
4. 如果 taskCh 缓冲区满（64），`b.taskCh <- func(){}` 也会永久阻塞

**修复方案**：
```go
// bridge.go - Submit 增加 context 支持
func (b *COMBridge) Submit(ctx context.Context, task func() error) error {
    if windows.GetCurrentThreadId() == b.threadID {
        return task()
    }
    errCh := make(chan error, 1)
    wrapped := func() { errCh <- task() }
    
    select {
    case b.taskCh <- wrapped:
    case <-ctx.Done():
        return ctx.Err()
    }
    
    select {
    case err := <-errCh:
        return err
    case <-ctx.Done():
        return ctx.Err()
    }
}

// Run 退出前排空已入队任务并标记失败
func (b *COMBridge) Run(ctx context.Context) {
    // ... 现有初始化代码 ...
    defer func() {
        // 排空 taskCh，让等待的 Submit 收到错误
        for {
            select {
            case task := <-b.taskCh:
                _ = task // 不执行，但 errCh 不会被写入
                // 需要更好的机制：用 wrapper struct 包含 errCh
            default:
                return
            }
        }
    }()
    // ... 主循环 ...
}
```

### C7. 普遍存在的 VARIANT 泄漏

|------|------|

| archiver.go | ~第232行 | `restrictedVar` | 无 Clear |
| archiver.go | ~第249行 | `countVar` | 无 Clear |

| reorganizer.go | ~第255行 | `itemsVar` | 无 Clear |

// 统一模式：获取 VARIANT 后立即 defer Clear
restrictedVar, err := comutil.SafeCallMethod(items, "Restrict", filter)
if err != nil {
    return res, fmt.Errorf("Restrict 过滤失败: %w", err)

restricted := restrictedVar.ToIDispatch()

```go
// comutil 包新增
func GetDispatchFromCall(disp *ole.IDispatch, method string, args ...interface{}) (*ole.IDispatch, func(), error) {
    v, err := SafeCallMethod(disp, method, args...)
    if err != nil {
        return nil, nil, err
    }
    d := v.ToIDispatch()
    cleanup := func() {
        SafeRelease(d)
        v.Clear()
    }
    return d, cleanup, nil
}
```

### M1. 配置热加载存在数据竞争（data race）

| 属性 | 内容 |
|------|------|
| **来源** | 报告1 M9、报告3 M6 |
| **位置** | [scheduler.go:55-61](file:///d:/Document/GO/outlookTool/internal/scheduler/scheduler.go#L55-L61) |
| **核查结论** | ✅ 属实 |

**问题描述**：`ReloadConfig` 通过 `*s.cfg = *newCfg` 就地更新共享 `*Config`，而 `Archiver`/`Reorganizer` 持有的同一指针在 COM 线程无锁读取字段（`RetainDays`、`MaxBatchSize` 等），属典型 data race。

**修复方案**：
```go
// scheduler.go - 使用 atomic.Value 存储 config
type Scheduler struct {
    cfgVal atomic.Value // 存储 *config.Config
    // ...
}

func (s *Scheduler) getConfig() *config.Config {
    return s.cfgVal.Load().(*config.Config)
}

func (s *Scheduler) ReloadConfig(path string) error {
    newCfg, _, err := config.LoadConfig(path) // 锁外加载
    if err != nil { return err }
    if err := config.ValidateConfig(newCfg); err != nil { return err }
    s.cfgVal.Store(newCfg) // 原子替换
    return nil
}
```

## 五、架构层面优化建议

### 5.1 提取 `internal/app` 生命周期管理包

`main.go` 当前承担了所有初始化和清理职责。建议提取 `App` 结构体封装启动/关闭流程，统一退出路径（托盘退出和 Web 退出走同一条路径）。

### 5.2 补充单元测试

整个项目无 `_test.go` 文件。以下纯函数适合优先补测：
- `router.CalcQuarter`
- `config.ValidateConfig`
- `BuildRestrictFilter`
- `parseTime`
- `CalcCutoffTime`

### 5.3 统一错误处理模式

当前存在三种错误处理方式混用：
- 返回 error 让调用方处理
- 内部 `logger.Error` 后继续
- `_ = err` 忽略错误

建议建立统一策略：对于 COM 操作结果必须检查并记录；对于辅助操作（如日志写入）可静默但应有计数。
