## 1. 项目背景与目标

### 1.1 业务背景

当前公司为员工配置的 Exchange 邮箱容量存在严格限制（如 1GB）。现有的归档插件存在严重的业务逻辑 Bug（如错误地将邮件归档至未来年份的 PST 文件中），导致大量历史邮件堆积在服务器端（OST），引发邮箱爆仓、收发故障及邮件查找困难。

### 1.2 项目目标

使用 Golang 开发一个轻量级、无感知的 Windows 客户端后台守护程序。通过调用微软原生的 COM 接口接管邮件归档工作。该程序需具备以下核心特性：

- **高稳定性**：支持长时间无人值守运行，不因异常中断
- **防多开**：系统级单例保护，防止资源争抢
- **极低资源占用**：CPU 和内存占用率可忽略不计
- **时间准确性**：严格按照邮件的**实际发生时间**（而非系统时间），按"当前季度"自动路由和移动邮件

## 2. 运行环境与技术选型

## 3. 核心功能与业务逻辑

### 3.1 进程管理与探活

#### 3.1.1 单例锁 (Mutex)

程序启动时必须通过 Windows 系统级命名 Mutex（推荐命名：`Global\OutlookAutoArchiver_{UserSID}`），确保同一台机器同一用户只能运行一个实例。若检测到已有实例运行，应弹出提示后静默退出。

#### 3.1.2 定时器触发

按配置文件设定的间隔（默认每 10 分钟）轮询执行一次归档任务。

#### 3.1.3 前置探活

每次轮询前，必须先扫描系统进程列表，确认 `OUTLOOK.EXE` 正在前台运行：

- **若未运行**：记录日志，直接跳过本次任务，进入休眠等待下一个轮询周期。**严禁**在 Golang 中隐式启动 Outlook 实例，防止产生不可见的幽灵进程锁定数据文件。
- **若已运行**：通过 `oleutil.GetActiveObject("Outlook.Application")` 获取当前实例的控制权。若获取失败（如 Outlook 处于启动/关闭过程中），记录错误日志并跳过本次任务。

### 3.2 动态季度路由与 PST 挂载校验

#### 3.2.1 时间基准（核心规则）

> ⚠️ **强制要求**：严格使用邮件自身的 `SentOn`（已发送邮件）或 `ReceivedTime`（收件箱）属性计算年份和季度，**严禁使用系统当前时间 `time.Now()`**。
>
> **反面案例**：3月31日 23:55 发送的邮件，程序在4月1日 00:05 执行移动，若按系统时间会错误分到 Q2，导致时间归档错乱。

#### 3.2.2 PST 命名规范

- 文件路径格式：`{pst_root_path}\{Year}_Q{Quarter}.pst`，例如 `D:\OutlookArchives\2024_Q2.pst`
- Outlook 侧边栏显示名称：`{Year}_Q{Quarter}`，例如 `2024_Q2`

#### 3.2.3 防重挂载检查

检查目标 PST 是否已挂载时：

- **判断依据**：遍历 Outlook 的 `Application.Session.Stores` 集合，比对 `Store.FilePath` 物理路径是否一致。**严禁**仅通过侧边栏显示名称判断——用户可能手动修改显示名称导致误判。
- 若文件路径匹配，视为已挂载，直接复用该 Store。
- 若未挂载，调用 `Application.Session.AddStore` 动态挂载。
- 若 PST 文件不存在，`AddStore` 会自动创建新文件。

### 3.3 自动化归档执行 (Move 操作)

#### 3.3.1 目标文件夹范围

程序需要支持**默认文件夹**和**用户自定义文件夹**的全量归档，确保邮箱内所有文件夹的邮件都能被正确同步到 PST 中。

**A. 默认文件夹（通过 COM 枚举值获取）**

| 文件夹     | COM 枚举值            | 时间属性       |
| ---------- | --------------------- | -------------- |
| 收件箱     | `GetDefaultFolder(6)` | `ReceivedTime` |
| 已发送邮件 | `GetDefaultFolder(5)` | `SentOn`       |

**B. 自定义文件夹（通过路径名称遍历获取）**

用户在 Outlook 中创建的自定义文件夹（如 `项目A`、`客户跟进/重要` 等）同样需要被归档。获取方式：

- 通过 `Application.Session.DefaultStore` 获取用户主邮箱的根 Store
- 通过 `Store.GetRootFolder()` 获取根文件夹对象
- 递归遍历 `Folder.Folders` 集合，获取所有子文件夹（包括嵌套的多级子文件夹）
- 跳过系统保留文件夹（如 `已删除邮件`、`垃圾邮件`、`草稿`、`发件箱`、`同步问题` 等），可通过配置文件中的排除列表控制

**C. 自定义文件夹的时间属性**

自定义文件夹中的邮件统一使用 `ReceivedTime` 属性作为时间基准。若 `ReceivedTime` 为空（极少数异常情况），回退使用 `CreationTime` 属性。

#### 3.3.2 延迟与过滤机制

- **延迟处理**：只处理时间戳在 X 分钟之前（默认 10 分钟，可配置）的邮件，防止移动正在与 Exchange 同步中的邮件导致 RPC 锁死报错。
- **条件查询优化**：**必须**使用 `Items.Restrict(Filter)` 进行前置过滤，**禁止**在 Go 代码中对整个文件夹执行全量 `for` 循环遍历。
  - 对于已发送邮件，使用 `SentOn` 过滤：
    ```
    [SentOn] < '2024-06-30 23:50:00' AND [MessageClass] = 'IPM.Note'
    ```
  - 对于收件箱及自定义文件夹，使用 `ReceivedTime` 过滤：
    ```
    [ReceivedTime] < '2024-06-30 23:50:00' AND [MessageClass] = 'IPM.Note'
    ```
- **类型过滤**：严格校验 `MessageClass` 属性，仅处理 `IPM.Note` 类型的常规邮件，跳过会议邀请 (`IPM.Schedule.Meeting.*`)、任务 (`IPM.Task`)、联系人 (`IPM.Contact`) 等特殊对象。

#### 3.3.3 执行移动与过载保护

- 调用 `MailItem.Move(targetFolder)` 将邮件物理转移至本地 PST 中对应季度文件夹。
- **批次限制**：单次任务限制最大处理数量（默认 500 封，可配置），超过部分留待下一个轮询周期处理，防止瞬间资源占用过高导致 Outlook 假死或触发 Exchange RPC 限流。
- 每完成一封邮件的移动后，应短暂 Sleep（如 50~100ms），给予 Outlook 和 MAPI 引擎喘息时间。
- **模拟运行模式 (dry_run)**：开启后，程序完整执行 COM 连接、邮件扫描、时间计算、目标 PST 路由等所有流程，但在最终执行 `MailItem.Move()` 时**仅输出日志而不实际移动**。适用于首次部署时验证配置正确性，以及全量整理前的预览检查。日志应打印：邮件主题、时间属性、源文件夹、目标 PST。

### 3.4 PST 内部文件夹结构

归档到 PST 后的邮件必须**完整镜像源邮箱的文件夹层级结构**，确保用户在 PST 中能以相同的目录方式查找邮件：

```
2024_Q2.pst（PST 根）
├── 收件箱/
│   └── (归档的收件箱邮件)
├── 已发送邮件/
│   └── (归档的已发送邮件)
├── 项目A/
│   └── (归档的项目A邮件)
├── 客户跟进/
│   ├── 重要/
│   │   └── (归档的邮件)
│   └── 待处理/
│       └── (归档的邮件)
└── ...
```

- PST 中的文件夹名称必须与源邮箱中的文件夹名称**完全一致**
- 多级嵌套的自定义文件夹，需在 PST 中递归创建相同的父子层级
- 若目标子文件夹不存在，程序应自动创建；若已存在则直接复用

### 3.5 全量整理功能（Full Reorganize）

用户可通过托盘右键菜单触发「全量整理」操作。该功能用于一次性清理邮箱积压，并**纠正所有已归档 PST 中的错乱邮件**（包括本工具和第三方工具产生的归档文件）。

#### 3.5.1 执行流程

```
用户点击「全量整理」
    │
    ▼
① 暂停后台自动归档定时任务
    │
    ▼
② 阶段一：归档邮箱中的邮件（与日常归档逻辑一致，但不限制批次数量）
    │
    ▼
③ 阶段二：扫描所有已有 PST 文件，纠正错乱邮件
    │
    ▼
④ 全量整理完成，恢复后台自动归档定时任务
```

#### 3.5.2 阶段一：全量归档邮箱邮件

- 执行与 §3.3 完全相同的归档逻辑（文件夹遍历、类型过滤、时间基准等）
- **不限制 `max_batch_size`**：处理全部符合条件的邮件，不分批
- **保留 `move_interval_ms` Sleep 特性**：防止 Outlook 假死
- **不启用延迟过滤 `safe_delay_minutes`**：全量整理时处理所有历史邮件，无需安全延迟

#### 3.5.3 阶段二：扫描已有 PST 并纠正错乱邮件

本阶段是全量整理的核心差异点。程序需要遍历所有已归档的 PST 文件，检查每一封邮件是否处于正确的季度 PST 中，若不正确则移动到正确的位置。

**A. 扫描范围**

程序需要扫描以下两类 PST 文件：

1. **本工具生成的 PST**：位于 `pst_root_path` 目录下，命名格式为 `{Year}_Q{Quarter}.pst`
2. **第三方工具生成的 PST**：自动遍历 `legacy_pst_scan_paths` 配置的目录，扫描其中所有 `.pst` 文件，排除本工具命名格式（`{Year}_Q{Quarter}.pst`）的文件后，剩余的均视为第三方 PST

> ℹ️ **为什么自动遍历**：第三方归档工具的 PST 命名规则不可预知（如 `2025Season3.pst`、`Archive2024.pst`、`备份.pst` 等），手动配置文件列表容易遗漏。通过目录扫描 + 排除本工具文件的方式，可确保不遗漏任何第三方 PST。

**B. 纠偏逻辑**

对扫描范围内的每个 PST 文件：

1. 若 PST 未挂载到 Outlook，先通过 `AddStore` 临时挂载
2. 递归遍历 PST 内所有文件夹，对每一封 `IPM.Note` 类型的邮件：
   - 读取邮件的 `SentOn`（已发送文件夹）或 `ReceivedTime`（其他文件夹）属性
   - 计算该邮件应归属的正确年份和季度
   - 若当前所在 PST 与正确的目标 PST（`{Year}_Q{Quarter}.pst`）**不一致**，则将邮件移动到正确的 PST 对应文件夹中
3. 移动时同样保留 `move_interval_ms` Sleep 特性
4. 每封邮件的纠偏操作需记录详细日志：原 PST → 目标 PST、邮件主题、时间属性值

**C. 第三方 PST 适配**

> ⚠️ **核心场景**：用户之前使用的第三方归档工具可能采用各种不同的命名规则（如 `2025Season3.pst`、`Archive_2024.pst`、`备份.pst` 等），且其中可能存在被错误归档的邮件（如 2026 年的邮件被归入 `2025Season3.pst`）。

处理规则：

- 程序自动遍历 `legacy_pst_scan_paths` 配置的目录，发现所有非本工具命名格式的 `.pst` 文件
- 第三方 PST 中的**所有邮件**（无论时间是否正确）均强制迁移到本工具定义的正确 PST 文件中（`{Year}_Q{Quarter}.pst`），以便统一管理
- 移动时保持目标 PST 内的文件夹层级结构与源文件夹一致（镜像规则同 §3.4）
- 迁移完成后，第三方 PST 保持挂载状态，**不自动卸载或删除**，由用户自行确认后手动处理
- 日志记录每封邮件的迁移详情：源 PST 文件名 → 目标 PST、邮件主题、时间属性值

#### 3.5.4 状态反馈与互斥保护

- 全量整理执行期间，托盘图标应切换为「整理中」状态图标（如旋转或脉冲动画效果）
- 托盘 Tooltip 显示当前进度：`全量整理中... 阶段X/2 - 已处理 N 封，已纠偏 M 封`
- 执行期间，`立即执行一次` 和 `全量整理` 菜单项置灰禁用，防止重复触发
- 全量整理完成后：
  - 托盘图标恢复正常状态
  - 弹出系统通知：`全量整理完成，共处理 X 封邮件，纠偏 Y 封错乱邮件`
  - 自动恢复后台定时任务

## 4. 关键研发规范与"避坑"指南

> ⚠️ **R&D 必读**：为保障程序长年稳定运行，研发在编写 Golang COM 交互代码时，**必须严格遵守**以下规范。

### 4.1 禁止直接文件 I/O

PST 文件是复杂的结构化数据库（基于 MS-PST/MS-OST 格式）。**严禁**使用 Go 的 `os` 或 `io` 标准库对正在使用的 `.pst` 文件进行任何形式的读写、移动或删除操作。所有数据操作必须 100% 通过 COM MAPI 接口执行。

### 4.2 强制锁定系统线程 (Thread Affinity)

Windows COM（STA 模式）严重依赖线程亲和性。在执行 COM 操作的 Goroutine 中，必须在入口处执行以下代码：

```go
runtime.LockOSThread()
defer runtime.UnlockOSThread()
```

否则 Go 的 M:N 协程调度器会将 Goroutine 迁移到其他系统线程，导致 COM 调用随机 Panic 或返回 `RPC_E_WRONG_THREAD` 错误。

### 4.3 极其严格的内存回收 (Release)

COM 对象不受 Go 的 GC 管理。生成的**每一个** `IDispatch` 指针（包括 Application、Namespace、Folder、Items，尤其是每一次遍历出来的 MailItem 及其属性对象），必须在用完后**立即**调用 `.Release()`。

**关键约束**：

- **禁止**将 `Release` 堆积在大型 `for` 循环外的 `defer` 中——这会导致循环结束前所有 COM 对象都不释放，引发严重的内存泄漏和句柄耗尽。
- 必须在循环体内部，每处理完一封邮件后立即释放该邮件相关的所有 COM 对象。

### 4.4 集合遍历防越界（移动操作核心法则）

当使用 `MailItem.Move()` 时，邮件会从当前文件夹的 `Items` 集合中被移除，集合的 `Count` 和索引会实时变化。

- **正序遍历（1 to Count）的致命问题**：移动第一个元素后，原来的第 2 个元素变成了第 1 个，但循环索引已指向 2，导致跳跃漏处理，最终可能越界 Panic。
- **强制要求**：必须使用**倒序遍历（从 Count downto 1）**，确保已处理元素的移除不影响未处理元素的索引。

### 4.5 接口兼容性与防御性编程

- 优先使用 `CallMethod` 并处理 `error` 返回值，避免大面积使用 `MustCallMethod`。
- 遇到移动失败的单封邮件（如邮件损坏、被其他进程锁定、网络瞬断），应**记录详细日志并跳过**，不能中断整个批处理任务。
- 对 COM 返回的 `HRESULT` 错误码进行分类处理，区分可重试错误和不可恢复错误。

## 5. 错误处理与恢复策略

### 5.1 异常分级

| 等级 | 场景                                   | 处理策略                                               |
| ---- | -------------------------------------- | ------------------------------------------------------ |
| 致命 | COM 初始化失败、Mutex 创建失败         | 记录日志，程序退出                                     |
| 严重 | Outlook 连接断开、PST 挂载失败         | 记录日志，跳过本次轮询，下次重试                       |
| 警告 | 单封邮件移动失败、PST 磁盘空间不足警戒 | 记录日志，跳过该邮件/发出警告，继续处理其余邮件         |
| 信息 | 正常轮询、无新邮件需处理               | 记录简要日志                                           |

### 5.2 磁盘空间监控

每次轮询前检查 PST 存储目录的可用磁盘空间：

- 低于 1GB 时：写入警告日志，托盘图标显示警告提示
- 低于 500MB 时：写入错误日志，暂停归档操作，托盘图标显示错误提示

### 5.3 PST 文件大小监控

单个 PST 文件不应超过 20GB（ANSI 格式限制为 2GB，Unicode 格式建议不超过 20GB）。当 PST 文件接近上限时，应记录警告日志提醒管理员介入。

## 6. 交互、配置与部署要求

### 6.1 部署形式

- 无界面后台进程 (Headless)，配合 Windows 系统托盘图标 (System Tray)。
- 托盘图标功能：
  - 左键双击：弹出状态摘要窗口（上次轮询时间、处理邮件数、当前 PST 信息等）
  - 右键菜单：
    - `立即执行一次`：手动触发一次常规归档（受 `max_batch_size` 限制）
    - `全量整理`：触发全量整理操作（详见 §3.5），执行期间此菜单项置灰
    - `打开日志目录`：快速定位日志文件
    - `打开配置文件`：用默认编辑器打开 `config.yaml`
    - `开机自启`：勾选/取消注册表自启项
    - `退出`：优雅关闭程序，释放所有 COM 资源（若全量整理正在执行，需先确认是否中断）
- 提供开机自启配置（写入注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`）。

### 6.2 配置文件 (`config.yaml`)

使用 YAML 格式保存配置文件，允许用户/管理员自定义以下参数：

```yaml
# PST 文件存储根目录
pst_root_path: "D:\\OutlookArchives"

# 轮询间隔（分钟）
poll_interval_minutes: 10

# 安全延迟时间（分钟），仅处理该时间之前的邮件
safe_delay_minutes: 10

# 单次最大处理邮件数（所有文件夹共享此额度）
max_batch_size: 500

# 归档模式：
#   "all"  - 归档所有文件夹（默认文件夹 + 用户自定义文件夹），排除 exclude_folders 中列出的
#   "list" - 仅归档 include_folders 中明确列出的文件夹
archive_mode: "all"

# 当 archive_mode 为 "all" 时生效：需要排除的文件夹名称列表
# 以下系统文件夹会被自动排除，无需手动添加：已删除邮件、垃圾邮件、草稿、发件箱、同步问题
exclude_folders:
  - "RSS 订阅"
  - "对话历史记录"

# 当 archive_mode 为 "list" 时生效：仅归档以下明确列出的文件夹
# 支持默认文件夹标识和自定义文件夹路径（多级用 / 分隔）
include_folders:
  - "Inbox"               # 收件箱
  - "SentItems"            # 已发送邮件
  - "项目A"                # 一级自定义文件夹
  - "客户跟进/重要"        # 多级自定义文件夹

# 日志保留天数
log_retention_days: 7

# 单封邮件移动后的间隔（毫秒），防止过快导致 Outlook 卡顿
move_interval_ms: 50

# 模拟运行模式（调试用）
# 开启后程序会执行完整的扫描、计算、路由流程，但不会真正执行 Move 操作
# 仅在日志中打印“将会移动”的详情，适用于首次部署验证或全量整理前预览
dry_run: false

# ==================== 全量整理相关配置 ====================

# 第三方 PST 扫描目录列表
# 全量整理时会自动遍历这些目录下的所有 .pst 文件
# 排除本工具命名格式（{Year}_Q{Quarter}.pst）的文件后，剩余的均视为第三方 PST
# 第三方 PST 中的所有邮件将被强制迁移到本工具的正确季度 PST 中
legacy_pst_scan_paths:
  - "D:\\OutlookArchives"       # 主归档目录（也是 pst_root_path）
  - "C:\\Users\\user\\Documents\\Outlook Files"   # Outlook 默认 PST 目录，可能存在其他工具的 PST
```

### 6.3 日志要求

由于程序运行在后台，必须引入滚动日志组件（推荐 `lumberjack` + `zap`）。

- **记录内容**：
  - 进程启动/退出事件
  - Outlook 探活状态（运行中/未运行）
  - 每次轮询：匹配到的邮件总数、成功移动数量、失败数量及详情、耗时
  - PST 挂载/创建事件
  - COM 错误码与异常堆栈
  - 磁盘空间警告
- **轮转策略**：日志文件按天轮转，单文件最大 50MB，保留最近 7 天（可配置），方便运维人员排查。
- **日志路径**：默认存储在程序同目录下的 `logs/` 文件夹中。
