### 🔍 深度审查分析报告

#### 第一次审查：业务逻辑与功能闭环 (Business Logic)
1. **归档范围的局限性**：文档当前仅明确了归档“已发送邮件(GetDefaultFolder(5))”。但在实际业务中，1GB 的容量限制通常是由收件箱（Inbox）和已发送邮件共同消耗的。建议在配置文件中支持配置需要归档的文件夹。
2. **跨季度时间边界问题**：文档提到“根据系统当前时间或邮件的 SentOn 属性计算”。**绝对不能使用系统时间**。例如：3月31日23:55发送的邮件，程序在4月1日00:05执行移动，如果按系统时间会分到 Q2，导致时间错乱。必须强制使用邮件的 `SentOn` 或 `ReceivedTime` 属性。
3. **PST 挂载防重逻辑**：`AddStore` 动作需要更加严谨。如果 PST 文件已经挂载，但用户在 Outlook 里手动把侧边栏的显示名称改了，按名称检查会失效导致重复挂载（甚至报错）。应遍历现有的 `Stores`，通过物理文件路径 (`Store.FilePath`) 来判断是否已挂载。

#### 第二次审查：技术架构与稳定性 (Technical Architecture)
1. **COM 集合操作的核心大坑**：在 COM 操作中，移动 (Move) 操作等同于从当前集合 (Items) 中移除元素。**如果使用正序遍历（1 to Count），移动第一个元素后，所有元素的索引会前移，导致跳跃漏处理并最终引发越界 Panic。** 必须在研发规范中强制要求：**倒序遍历（从 Count downto 1）**。
2. **性能优化**：如果“已发送”文件夹有 1 万封邮件，每次轮询都遍历 1 万次会卡死 Outlook。不应在 Golang 里进行全量循环，必须强制使用 `Items.Restrict()` 或 `Items.Find()` 让 Outlook MAPI 引擎在底层先过滤出“10分钟前”的邮件，然后再遍历结果集。
3. **单例运行锁 (Mutex)**：既然是守护进程，必须防止用户多次双击执行导致启动多个实例，产生资源争抢。需要在 Golang 启动时检查全局 Mutex。

#### 第三次审查：部署、运维与边缘场景 (Ops & Edge Cases)
1. **日志记录缺失**：作为无界面守护进程（Headless），一旦出问题排查将是灾难。必须加入本地滚动日志机制（如 Lumberjack），记录每次轮询的移动数量、耗时、失败原因。
2. **会话隔离问题 (Session 0)**：文档提到守护进程。需明确指出**不能将其注册为 Windows Service**。因为 Windows 服务运行在 Session 0，而 Outlook 运行在用户的 Session 1+。跨会话调用 COM 会失败或引发安全警告。必须通过注册表 `HKCU\...\Run` 或计划任务（仅用户登录时）启动。
3. **过载保护 (Throttling)**：如果用户休假 2 周回来，积压了大量邮件，一次性移动可能导致 Outlook 假死或触发 Exchange RPC 限制。需在配置文件中增加“单次最大处理量（Batch Size）”。

# 需求说明书：Outlook 自动归档守护进程 (Outlook Auto-Archiver)

## 1. 项目背景与目标
### 1.1 业务背景
当前公司为员工配置的 Exchange 邮箱容量存在严格限制（如 1GB）。现有的归档插件存在严重的业务逻辑 Bug（如错误地将邮件归档至未来年份的 PST 文件中），导致大量历史邮件堆积在服务器端（OST），引发邮箱爆仓、收发故障及邮件查找困难。

### 1.2 项目目标
使用 Golang 开发一个轻量级、无感知的 Windows 客户端后台守护程序。通过调用微软原生的 COM 接口接管邮件归档工作。该程序需具备高稳定性、防多开、极低的资源占用，并能准确无误地按照邮件的**实际发生时间**，按“当前季度”自动路由和移动邮件。

## 2. 运行环境与技术选型
* **开发语言**：Golang (编译为单文件二进制无依赖执行)。
* **核心依赖库**：`github.com/go-ole/go-ole` (用于 Windows COM 接口调用)。
* **目标操作系统**：Windows 10 / Windows 11。
* **依赖软件环境**：Microsoft Office LTSC 2024 专业增强版 (必须为经典版 Classic Outlook，**不支持**基于 WebView2 的“新版 Outlook”)。
* **运行权限**：普通用户权限运行（User Session），**禁止部署为 Windows 服务 (Session 0)** 以避免跨会话 COM 调用失败。

## 3. 核心功能与业务逻辑
### 3.1 进程管理与探活
* **单例锁 (Mutex)**：程序启动时必须通过系统级 Mutex 或锁文件，确保同一台机器同一用户只能运行一个实例。
* **定时器触发**：按配置文件设定的间隔（如每 10 分钟）轮询一次。
* **前置探活**：必须先扫描系统进程列表，确认 `OUTLOOK.EXE` 正在前台运行。
  * **若未运行**：直接跳过本次任务，进入休眠。（**严禁**在 Golang 中隐式启动 Outlook 实例，防止产生不可见的幽灵进程锁定数据文件）。
  * **若已运行**：通过 `oleutil.GetActiveObject("Outlook.Application")` 获取当前实例的控制权。

### 3.2 动态季度路由与 PST 挂载校验
* **时间基准**：严格使用邮件自身的 `SentOn` (已发送) 或 `ReceivedTime` (收件箱) 属性计算年份和季度，**严禁使用系统当前时间**，防止跨季度时数据归档错乱。
* **命名规范**：生成目标数据文件路径，如 `C:\...\Outlook Files\2024_Q2.pst`，并在 Outlook 侧边栏显示名称为 `2024_Q2`。
* **防重挂载检查**：检查目标 PST 是否已挂载。
  * **判断依据**：遍历 Outlook 的 `Application.Session.Stores`，比对 `Store.FilePath` 物理路径是否一致，而非单纯比对侧边栏名称。
  * 若未挂载，调用 `Application.Session.AddStore` 动态挂载。

### 3.3 自动化归档执行 (Move 操作)
* **目标文件夹范围**：默认支持“已发送邮件(GetDefaultFolder(5))”和“收件箱(GetDefaultFolder(6))”。
* **延迟与过滤机制**：
  * **延迟处理**：只处理时间戳在 X 分钟之前（如 10 分钟前）的邮件，防止移动正在与 Exchange 同步中的邮件导致 RPC 锁死报错。
  * **条件查询优化**：必须使用 `Items.Restrict(Filter)` 进行前置过滤，**禁止在 Go 代码中对整个文件夹执行全量 `for` 循环遍历**。
  * **类型过滤**：严格校验 `MessageClass` 属性，仅处理 `IPM.Note` 类型的常规邮件，跳过会议邀请 (`IPM.Appointment`) 等特殊对象。
* **执行移动与过载保护**：调用 `MailItem.Move(targetFolder)` 将邮件物理转移至本地 PST。单次任务需限制最大处理数量（如 500 封），超过部分留待下一个轮询周期处理，防止瞬间资源占用过高导致 Outlook 假死。

## 4. 关键研发规范与“避坑”指南 (R&D 必读 ⚠️)
为保障程序长年稳定运行，研发在编写 Golang COM 交互代码时，**必须严格遵守**以下规范：

1. **绝对禁止的文件 I/O 操作**：PST 文件是复杂的结构化数据库。严禁使用 Go 的 `os` 或 `io` 标准库对正在使用的 `.pst` 文件进行读写或移动。所有操作 100% 通过 COM MAPI 接口执行。
2. **强制锁定系统线程 (Thread Affinity)**：Windows COM 严重依赖线程亲和性。在执行 COM 操作的 Goroutine 顶部，必须执行 `runtime.LockOSThread()` 并用 defer 解锁，否则 Go 的协程调度会导致程序随机 Panic。
3. **极其严格的内存回收 (Release 炼狱)**：COM 对象不受 Go 的 GC 管理。生成的**每一个** `IDispatch` 指针（包括 Application, Namespace, Folder, Items, 尤其是每一次遍历出来的 MailItem 及其属性），必须在用完后立即调用 `.Release()`。**切勿将 Release 堆积在大型 for 循环外的 defer 中**，必须在循环体内及时释放，否则会导致严重的内存泄漏和句柄耗尽。
4. **集合遍历防越界（移动操作的核心法则）**：当你使用 `MailItem.Move()` 时，邮件会从当前文件夹的 `Items` 集合中消失。如果使用正序遍历，索引会发生变化导致漏处理或越界崩溃。**必须通过索引倒序遍历（从 Count downto 1）**。
5. **接口兼容性防御**：建议使用 `CallMethod` 并处理 error 返回值，避免大面积使用 `MustCallMethod`。遇到移动失败的单封邮件（如损坏或被锁定），应记录日志并跳过，不能中断整个批处理任务。

## 5. 交互、配置与部署要求
### 5.1 部署形式
* 无界面后台进程 (Headless)，带有简单的 Windows 托盘图标 (System Tray) 最佳，提供“手动触发一次”和“退出”功能。
* 提供开机自启配置引导（建议写入注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`）。

### 5.2 配置文件 (`config.yaml`)
使用yaml格式保存配置文件，以下为json举例示意
允许用户/管理员自定义以下参数：
```json
{
  "pst_root_path": "D:\\OutlookArchives", 
  "poll_interval_minutes": 10,
  "safe_delay_minutes": 10,
  "max_batch_size": 500,
  "target_folders": ["SentItems", "Inbox"],
  "log_retention_days": 7
}
```

### 5.3 日志要求
由于程序运行在后台，必须引入滚动日志组件（如 `lumberjack` + `zap`）。
* **记录内容**：进程启动/退出、检测到 Outlook 状态、每次轮询匹配到的邮件数、成功移动数量、失败详情及 COM 错误码。
* 日志文件需按天轮转，保留最近 7 天，方便运维人员排查。
